# 人机协作报告
## 一、AI交互日志

本次出租车数据分析项目开发过程中，我多次使用AI工具辅助完成代码设计、问题排查以及项目结构优化。AI主要参与了项目框架设计、代码编写建议、错误分析和文档整理等工作。但是在实际使用过程中，我发现AI生成的内容并不是完全正确，需要结合运行结果不断修改和验证。

### 1. M1数据处理模块交互记录
#### （1）关键Prompt

“请根据课程要求完成纽约出租车Parquet数据的数据读取、清洗、异常值检测和特征工程，需要生成数据质量报告，并符合项目工程结构。”

AI输出摘要

AI给出了基于pandas和pyarrow的数据处理方案，主要包括：

使用pyarrow读取Parquet文件；
pandas转换DataFrame；
统计缺失率；
使用IQR方法检测异常值；
中位数填充缺失值；
构造小时、星期、高峰等时间特征。
我的采用情况

该部分整体采用，但是进行了部分修改。

主要修改：

将原始代码封装成：
def data_processing():

函数形式，使其能够被main.py调用。

修改文件路径。

AI最初生成代码时使用绝对路径，而课程要求使用相对路径，因此修改为：

DATA_PATH="data/yellow_tripdata_2026-01.parquet"
增加输出目录：
outputs/

用于保存：

data_quality_report.csv
cleaned_data_sample.csv
AI出现的问题

在第一次运行代码时出现：

ModuleNotFoundError:
No module named 'pyarrow'

AI建议直接安装pyarrow，但是安装后仍然无法运行。

经过检查发现问题不是代码错误，而是PyCharm使用了错误的Python解释器：

[invalid] Python 3.13

最后重新配置虚拟环境解释器后解决。

经验总结

通过这个问题，我发现AI能够快速定位代码层面的错误，但是对于开发环境问题，需要结合IDE实际情况判断，不能完全依赖AI给出的解决方案。

### 2. M2可视化模块交互记录
#### （1）关键Prompt

“请根据出租车数据分析要求完成四项可视化，包括时间需求规律、区域热度分析、费用影响因素分析，并增加geopandas区域分级设色地图。”

AI输出摘要

AI生成了：

matplotlib绘制折线图；
柱状图分析TOP10区域；
散点图分析距离与费用关系；
geopandas读取taxi_zones.shp绘制地图。
我的采用情况

基础绘图部分基本采用。

但是区域地图部分进行了多次修改。

AI错误案例1：缺少mapclassify依赖

运行地图代码时出现：

ImportError:
The 'mapclassify' package is required

AI分析认为：

geopandas中的scheme="quantiles"
需要额外安装mapclassify

判断正确，因此安装：

pip install mapclassify

解决。

AI错误案例2：地图图例参数错误

AI生成：

legend_kwds={
"label":"上车订单量分级"
}

运行后出现：

TypeError:
Legend.__init__() got unexpected keyword argument 'label'

原因：

matplotlib新的legend接口不支持label参数。

修改为：

legend_kwds={
"title":"上车订单量分级"
}

问题解决。

AI错误案例3：分级数量错误

修改后继续运行：

ValueError:
Number of labels must match number of bins

原因：

quantiles分成5组，但是legend标签数量不匹配。

最终调整分级参数，使地图正常输出。

我的修改

legend_kwds={
            "title":"上车订单量分级"
        }

最终实现照片输出

AI错误案例

（1）夜间订单回答错误

用户输入：

为什么晚上出租车订单更多？

系统回答：

夜间订单数量约为691446条。

发现问题：

该问题实际需要的是原因解释，而不是简单统计。

修改：

对于包含“为什么”“原因”等关键词的问题，不直接返回统计结果，而调用大模型进行解释。

（2）DeepSeek接口调用问题

尝试接入大模型时出现：

原因：

没有配置API Key。

后来配置API Key

环境变量：

DEEPSEEK_API_KEY

同时发现：

直接将中文内容放入HTTP Header会出现：

## 二、三阶段代码对比

本项目选择“M4智能问答系统”作为典型功能进行比较。

### 1. Native版（完全独立编写）

实现方式：

通过if判断用户输入：

if "车费" in question:
    return fare

优点：

编写简单；
运行速度快；
不依赖外部服务。

缺点：

只能回答固定问题；
无法理解自然语言变化。

例如：

“多少钱”

和：

“平均费用是多少”

需要分别设置规则。

### 2. Prompt版（AI辅助）

实现方式：

通过向AI描述需求，让AI生成代码框架。

例如：

Prompt：

“生成出租车数据问答系统，支持区域、费用、模型评价查询。”

AI生成：

查询函数；
主循环；
调用结构。

优点：

开发效率明显提高；
可以快速搭建项目结构；
减少重复代码。

缺点：

生成代码需要人工检查；
对环境问题处理能力有限。
### 3. Vibe版（自然语言驱动）

实现方式：

用户直接输入：

“为什么晚上出租车需求较高？”

系统调用大模型进行解释。

优点：

交互方式更加自然；
能处理规则无法覆盖的问题；
具有一定推理能力。

缺点：

回答可能出现错误；
需要设计Prompt限制输出范围；
API依赖外部服务。
三种方式比较

|方式|效率|理解能力|
|-|-|-|
|Native|低|低|
|Prompt|较高|中|
|Vibe|最高|较强|

实际开发中，三者不是替代关系，而应该结合使用。

本项目最终采用： 规则查询 + AI解释 的混合方式。

## 三、项目反思

通过本次项目，我对AI工具的认识发生了一定变化。刚开始认为AI可以直接生成完整代码，只需要复制运行即可。但是在实际开发过程中发现，AI生成的代码仍然需要人工理解和验证。

例如，在项目开发中，AI能够快速生成数据处理、模型训练和可视化代码，但是在实际运行过程中出现了Python解释器错误、第三方库缺失、地图绘制参数错误等问题。如果完全依赖AI，而不理解代码运行逻辑，很难发现这些问题。

同时，我也认识到AI比较擅长提高开发效率，例如快速搭建代码框架、提供算法思路、解释报错原因。但是AI并不能完全替代开发者，它无法准确感知本地环境，也可能生成不符合实际需求的代码。因此，在使用AI工具时，需要保持人工判断，将AI作为辅助工具，而不是完全依赖。

通过本次人机协作开发，我认为未来的软件开发模式可能不是人与AI竞争，而是人与AI共同完成任务。人工负责需求分析、结果判断和最终决策，AI负责提高效率和提供技术支持，两者结合才能发挥更大的价值。
